## 快速开始

### 构思项目

在开始时，不需要完全想清楚项目的所有细节。你只需要先准备好以下信息：

* 这个项目要解决什么问题；
* 主要用户是谁；
* 你希望它提供哪些核心功能；
* 大致使用什么技术栈；
* 是否有明确的部署方式；
* 是否有必须遵守的约束。

然后让 Agent 阅读本仓库中的文档，并根据你的项目信息初始化项目结构。

### 让 Agent 初始化项目文档

可以使用如下提示词：

```markdown
请阅读当前仓库的 AGENTS.md 和 docs 目录。
我要基于这个模板创建一个新项目，项目信息如下： 
- 项目名称： 
- 项目目标： 
- 目标用户： 
- 核心功能： 
- 技术栈： 
- 部署方式： 
- 重要约束：

请先不要直接大量写代码，而是先根据模板完善项目文档，包括 README、架构说明、开发计划、测试策略和 Agent 工作指引。
```

### 开始开发

初始化完成后，不建议让 Agent 一次性完成整个项目。

更推荐的方式是：

1. 先定义一个小而明确的目标；
2. 让 Agent 写执行计划；
3. 人类确认设计方向；
4. Agent 实现 + 测试；
5. 人类检查关键取舍，并将经验更新回文档。

## 开发指引

开发是自由的，本模板只提供一套推荐工作方式。

### 起步阶段

起步阶段的重点在于让 Agent 理解项目并完善各处文档。推荐跟随 [快速开始](#快速开始)。

### 开发阶段

随着项目深入，代码量会逐渐增多。此时人类开发者不应尝试 review 所有代码细节，而应更多关注系统和架构层面的控制。

推荐做法：

* 采用深度优先的工作方式，把大目标拆成更小的设计、实现、测试、评审模块；
* 每完成一个 branch 或一个相对独立的小模块后，开启新的 Agent session；
* 在新 session 开始时，让 Agent 先阅读 AGENTS.md、相关 docs 和当前任务计划；（对于大部分 Agent，这应当是一个默认的过程）
* 人类负责确认架构方向、接口边界和产品取舍；
* Agent 负责具体实现、测试、局部重构和文档同步；
* 发现反复出现的问题后，优先更新文档、测试或工具，而不是只在聊天中提醒；
* 每周进行一次清理，集中处理技术债、过时文档、重复代码和架构漂移。

### 清理阶段

Agent-first 开发会提高代码产出速度，也会放大坏模式。

因此，建议周期性执行“清理”：

* 删除无用代码；
* 合并重复逻辑；
* 更新过时文档；
* 检查目录结构是否仍然合理；
* 检查测试是否覆盖关键路径；
* 检查 Agent 是否反复犯同类错误；
* 将新的约束写入 AGENTS.md 或 docs；
* 将可以机械检查的规则转化为 lint、test 或 CI。

## 开发者与 Agent 的分配

大多数文档中都包含了对人类开发者以及 Agent 编写工作的分配。

### 人类开发者更适合负责

* 项目愿景；
* 产品边界；
* 用户价值判断；
* 架构设计；
* 技术选型取舍；
* 安全与隐私底线；
* 关键接口设计；
* 验收标准；
* 是否接受技术债；
* 是否合并改动。

### Agent 更适合负责

* 根据文档生成初始代码；
* 实现局部功能；
* 编写测试；
* 修复明确 bug；
* 更新文档；
* 执行重构；
* 总结 diff；
* 检查代码风格；
* 根据失败日志定位问题；
* 根据 review 反馈修改 PR。

### 共同维护的内容

* AGENTS.md；
* 架构文档；
* 开发计划；
* 测试策略；
* 技术债列表；
* 决策记录；
* Changelog。
